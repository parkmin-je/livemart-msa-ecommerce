# 결제 금액 검증을 프론트엔드에만 했다가 CVSS 9.3 취약점을 만든 이야기

돌이켜보면 정말 부끄러운 코드다. 결제 금액을 프론트엔드에서 그냥 서버로 넘기고, 서버에서는 그 값을 그대로 믿었다. 당시에는 "어차피 우리가 만든 UI에서만 호출하는데 뭐가 문제야?"라고 생각했다.

## 처음 설계: 클라이언트가 금액을 결정했다

초기 설계는 이랬다. 사용자가 결제 버튼을 누르면 프론트엔드가 장바구니에 있는 상품들의 총 금액을 계산해서 결제 API에 넘겼다.

```json
POST /api/v1/payments
{
  "orderNumber": "ORD-20260315-A1B2C3",
  "userId": 42,
  "amount": 59000,
  "paymentMethod": "CREDIT_CARD"
}
```

서버는 이 요청을 받아서 그냥 결제 처리를 했다. `amount` 필드를 신뢰한 것이다. `PaymentRequest.Create`의 검증은 형식 유효성만 체크했다.

```java
public static class Create {
    @NotBlank private String orderNumber;
    private Long userId;
    @NotNull @DecimalMin("0.01") private BigDecimal amount;  // 0.01원 이상이면 통과
    @NotNull private PaymentMethod paymentMethod;
}
```

`@DecimalMin("0.01")` — 이게 전부였다. 금액이 0.01원 이상이기만 하면 통과다.

## "어, 이거 그냥 조작 가능하지 않나?"

어느 날 밤에 결제 플로우를 다시 살펴보다가 문득 이런 생각이 들었다. "내가 Burp Suite 같은 도구로 요청을 가로채서 `amount` 필드를 `1`로 바꾸면 어떻게 되지?"

직접 해봤다. `curl`로 직접 결제 API를 호출했다.

```bash
curl -X POST http://localhost:8082/api/v1/payments \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer [토큰]" \
  -d '{
    "orderNumber": "ORD-20260315-A1B2C3",
    "amount": 1,
    "paymentMethod": "CREDIT_CARD"
  }'
```

결과: `201 Created`. 59,000원짜리 주문이 1원에 결제됐다.

손이 떨렸다. 이건 단순한 버그가 아니라 결제 시스템의 근본적인 취약점이었다.

## 공격 시나리오를 구체적으로 생각해보니

실제 공격 시나리오는 이렇다.

1. 공격자가 정상적으로 주문을 생성한다. 장바구니에 100만원짜리 노트북을 담는다.
2. 결제 페이지에서 프론트엔드 코드를 확인하거나, 브라우저 개발자 도구나 Burp Suite로 네트워크 요청을 가로챈다.
3. `amount` 필드를 `1`로 수정해서 전송한다.
4. 서버는 1원 결제를 승인한다.
5. Saga 패턴에 의해 ORDER_CONFIRMED 이벤트가 발행되고, 재고가 차감된다.
6. 100만원짜리 노트북이 1원에 배송된다.

Toss Payments 같은 PG사를 쓴다면 PG사에서 금액 검증을 한 번 더 하지만, 내부 결제(시뮬레이션 결제나 기업 고객 후불 결제)에서는 이 로직을 그냥 통과한다.

CVSS 스코어로 따지면 9.3에 해당하는 심각한 취약점이다. 기밀성(Confidentiality) 영향은 없지만 무결성(Integrity)과 가용성(Availability) 영향이 모두 높고, 네트워크를 통해 인증된 사용자라면 누구나 공격 가능하다.

## 수정 방법: 서버에서 금액을 직접 조회한다

해결 방법은 명확했다. 클라이언트가 넘긴 금액을 믿지 말고, 서버에서 직접 order-service에 해당 주문의 실제 금액을 조회해서 비교하면 된다.

`payment-service`에 `OrderFeignClient`를 추가했다.

```java
@FeignClient(name = "order-service", path = "/api/orders")
public interface OrderFeignClient {

    @GetMapping("/number/{orderNumber}")
    OrderInfo getOrderByNumber(@PathVariable("orderNumber") String orderNumber);
}
```

그리고 `PaymentController`에서 결제 처리 전에 금액 검증을 수행한다.

```java
@PostMapping
public ResponseEntity<PaymentResponse> processPayment(@Valid @RequestBody PaymentRequest.Create request) {
    // 서버 측 주문 금액 검증 — 클라이언트 금액 조작 방어
    try {
        OrderInfo order = orderFeignClient.getOrderByNumber(request.getOrderNumber());

        if (order.getTotalAmount() != null &&
                order.getTotalAmount().compareTo(request.getAmount()) != 0) {
            log.warn("결제 금액 불일치 감지 - orderNumber: {}, 요청: {}, 실제: {}",
                    request.getOrderNumber(), request.getAmount(), order.getTotalAmount());
            throw new ResponseStatusException(HttpStatus.BAD_REQUEST, "결제 금액이 일치하지 않습니다");
        }

        if ("PAID".equals(order.getStatus()) || "PAYMENT_COMPLETED".equals(order.getStatus())) {
            throw new ResponseStatusException(HttpStatus.CONFLICT, "이미 결제 완료된 주문입니다");
        }
    } catch (ResponseStatusException e) {
        throw e;
    } catch (Exception e) {
        // order-service 호출 실패 시 결제 진행 차단 (Fail-Closed)
        log.error("주문 정보 조회 실패 - orderNumber: {}, error: {}", request.getOrderNumber(), e.getMessage());
        throw new ResponseStatusException(HttpStatus.SERVICE_UNAVAILABLE,
                "주문 정보를 확인할 수 없습니다. 잠시 후 다시 시도해주세요.");
    }

    return ResponseEntity.status(HttpStatus.CREATED).body(paymentService.processPayment(request));
}
```

여기서 중요한 설계 결정이 하나 있다. order-service 호출이 실패했을 때 결제를 허용할지, 차단할지다.

두 가지 선택지가 있다.
- **Fail-Open**: 조회 실패 시 결제를 진행. 가용성은 높지만 검증 우회 가능.
- **Fail-Closed**: 조회 실패 시 결제를 차단. 가용성은 낮지만 보안은 강화.

결제 시스템에서 잘못된 금액으로 결제되는 것보다 일시적 장애로 결제가 안 되는 게 낫다고 판단해서 **Fail-Closed**를 선택했다. 503을 반환하면 사용자가 재시도할 수 있다. 하지만 1원 결제는 재시도해도 계속 1원이다.

## 추가로 발견한 문제: 중복 결제

같은 주문 번호로 결제가 여러 번 시도되는 경우도 체크해야 했다. 위 코드에서 `order.getStatus()` 검사로 이미 결제 완료된 주문에 대한 중복 결제를 막고 있다. 여기에 더해 `@IdempotencyKey` 애노테이션을 결제 엔드포인트에 붙여서 같은 요청이 중복 처리되지 않도록 했다.

```java
@PostMapping
@IdempotencyKey(prefix = "payment-process")
@AuditLog(action = "PROCESS_PAYMENT", resource = "Payment")
public ResponseEntity<PaymentResponse> processPayment(...)
```

## Toss Payments의 경우는 다르다

`/toss/confirm` 엔드포인트는 조금 다르게 처리된다. Toss Payments는 PG사 자체적으로 금액 검증을 한다. Toss 위젯에서 결제를 시도할 때 등록된 금액과 승인 요청 금액이 다르면 Toss API 레벨에서 오류를 반환한다. 그래서 `/toss/confirm`에서는 별도의 금액 검증 없이 Toss API 응답을 신뢰한다.

근데 이것도 완전히 안심할 수는 없다. Toss 쪽에 문제가 생기거나 내부 결제로 전환할 때를 대비해서 서버 측 검증 로직은 항상 존재하는 게 좋다.

## 교훈

이 경험에서 얻은 교훈은 명확하다.

**클라이언트에서 온 데이터는 절대 신뢰하지 않는다.** 금액, 수량, 사용자 ID, 권한 — 전부 서버에서 직접 검증해야 한다. "어차피 우리 UI에서만 호출해"라는 생각은 API가 외부에 노출되는 순간 무너진다. 아니, 노출되지 않아도 브라우저 개발자 도구만 열면 충분히 조작 가능하다.

이건 OWASP Top 10에서 항상 상위권에 있는 "Broken Access Control"과 "Security Misconfiguration" 범주에 해당한다. 보안 교과서에 나오는 내용이지만 실제로 코드를 짜다 보면 쉽게 놓친다.

설계 단계에서 "이 데이터의 출처는 어디인가, 신뢰할 수 있는가"를 항상 질문해야 한다. 특히 돈과 관련된 로직에서는.
